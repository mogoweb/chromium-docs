When will a fix ship in Chrome (stable or canary)?

Here's a quick guide on how to track Blink changes as they make their way to releases of Chrome. 
Blink includes all HTML, CSS, DOM, web platform stuff, as well as Chrome DevTools.

On chromestatus.com?

First things first, chromestatus.com was built for this. If the feature is big, it will probably be there.
On the left you can see all features currently in beta. Those should be in stable soon.
You can search for other features and view what release they are Enabled by default.

Not on Chromestatus?

Alright, buckle up. This is fun stuff.
It all comes down to tracking the Blink revision number.
  • First, we're assuming you're looking at the bug somewhere in code.google.com/p/chromium/issues/
  • Now, when a commit lands, an automated comment is added to the Chromium bug.
  • All commits have a git hash and an incremental revision number.
  • You can see abhijeet's commit below has r192839. You can also see this is for the Blink repo.
    • The Chromium repo is actually separate (so far!) and it has its own set of numbers. Clicking on the r192839 link will actually go to the Chromium rev 192839... It's a bug. :) 
    • But! You can view the blink one with this URL: blinkrev.hasb.ug/192839



Waiting for it to hit Chrome Stable?

Generally it takes ~10 weeks from a commit landing to hit stable. 

  • This page shows the version numbers of all the current releases of Chrome.
  • But we need to know which rev of Blink is in those. For that we go to the more verbose view…
    • https://omahaproxy.appspot.com/
  • Look at the base_webkit_position column. It indicates the latest rev of Blink that's in that release. 
  • So if it's higher than the commit you're looking for, it's probably in that release.  

  • Look over at the current_version column to see if it's in Chrome 43 or 44…  Then it's just a matter of when your target version comes to stable.
  • Chrome ships a new major version to stable generally every 6 weeks, but in some cases we may choose to skip a release.
  • To figure out when your version will ship.. you could go to wikipedia.org/Google_Chrome_release_history , look for the release date of the current Stable channel and then calculate what's 6 or 12 or 18… weeks from then.



Waiting for it to land in Canary or an edge build?

You could use the above technique... but here's the guide for hackers or folks who don't want to wait for Canary or Dev Channel:

Does Chromium have the latest rev of Blink?

  • Want to see when this commit was rolled in Chromium. 
  • Blink comes in with a DEPS roll (dependencies). You can view all of those here: chromium.googlesource.com/chromium/src/+log/master/DEPS
  • Now you're looking for a Blink/WebKit DEPS roll that would include 192839
    • In our case,  it was this one (ba11baa)
    • Protip: V8, Chrome's JavaScript VM, is updated via DEPS rolls too, so this technique works for their repo/commits.
  • If the roll hasn't happened yet, you'll just have to be patient.
  • But in this case, we know it was merged into Chromium. Lets find a built that has it to try out.

Get a fresh Chromium build

  • Get the latest chromium in either of these two ways: Download Chromium
    • These builds are generated like every hour, so as long as Blink has rolled it, there's a good chance.
  • Canary is updated every day (at ~4am PST).
  • Either way, open up either and go to about:version 

  • Next to Blink is "537.36" (which you can ignore because it never changes anymore)...
  • But next to 537.36 is....  the Blink rev number.
  • As long as that rev number is higher than the one you're interested in... you're good!
    • (Of course this assumes no reverts of either the commit or the DEPS roll, but you'll probably have spotted those already)

Enjoy the fix or feature you've been waiting for!
